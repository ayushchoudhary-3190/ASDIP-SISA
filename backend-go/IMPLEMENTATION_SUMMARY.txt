AI Security Platform - Backend-Go Implementation Summary
=======================================================

This document explains the features implemented in the backend-go service
and how they work together to form the AI Gateway component of the platform.

OVERVIEW
--------
The backend-go service acts as the AI Gateway in the AI Security Platform.
It receives analysis requests from the frontend, validates them, forwards
them to the Python ML service for processing, and returns the results.

ARCHITECTURE
------------
The backend follows a modular structure:
- cmd/server/main.go: Application entry point and HTTP server setup
- internal/handlers/: Request handlers for different endpoint types
- internal/models/: Data structures for request/response payloads
- internal/gateway/: Communication with external services (ML service)
- internal/middleware/: Custom middleware components

FEATURES IMPLEMENTED
--------------------

1. HTTP SERVER (main.go)
   - Gin web framework for routing and middleware
   - Health check endpoint at GET /health
   - Analysis endpoints:
     * POST /analyze - For JSON requests (text, chat, sql)
     * POST /analyze/upload - For file uploads (logs, documents)
   - CORS middleware configured for frontend communication
   - Custom error handling middleware for panic recovery
   - Request/response logging
   - Graceful shutdown handling

2. REQUEST HANDLING (handlers/analyze.go)
   - AnalyzeHandler: Processes JSON analysis requests
     * Validates JSON input structure
     * Checks input type (text, file, sql, chat, log)
     * Forwards validated requests to ML service
     * Returns ML service response to client
   
   - AnalyzeUploadHandler: Processes file upload analysis
     * Extracts form data (input_type, options)
     * Validates input type
     * Reads uploaded file content
     * Creates analysis request with file content
     * Forwards to ML service
     * Returns ML service response

3. DATA MODELS (models/)
   - AnalyzeRequest: Input structure for analysis requests
     * InputType: Type of input (text|file|sql|chat|log)
     * Content: Actual content to analyze
     * Options: Analysis configuration (mask, block_high_risk, log_analysis)
   
   - AnalyzeResponse: Output structure from ML service
     * Summary: Text summary of findings
     * ContentType: Type of content analyzed
     * Findings: Array of detected security findings
     * RiskScore: Numeric risk score (0-20+)
     * RiskLevel: String risk level (low, medium, high, critical)
     * Action: Recommended action (allowed, masked, blocked)
     * Insights: AI-generated security insights
   
   - Finding: Individual security finding
     * Type: Category of finding (email, password, api_key, etc.)
     * Risk: Risk level of this finding
     * Line: Line number where found (for logs/files)
     * Value: The actual sensitive value found

4. ML SERVICE COMMUNICATION (gateway/)
   - MLClient: HTTP client for communicating with Python ML service
     * 30-second timeout for requests
     * JSON serialization/deserialization
     * Error handling for network issues and service errors
   
   - Gateway helpers: Singleton pattern for ML client
     * Initializes with ML_SERVICE_URL from environment
     * Defaults to localhost:8000 for development
     * Provides simple CallMLService function for handlers

5. MIDDLEWARE
   - ErrorHandler: Recovers from panics and returns 500 errors
   - CORS: Configured in main.go to allow frontend requests
   - Logger: Built-in Gin logger for request/response logging
   - Recovery: Built-in Gin recovery for panic handling

HOW IT WORKS TOGETHER
---------------------
1. Frontend sends request to /analyze or /analyze/upload
2. Gin router applies middleware (logging, error handling, CORS)
3. Handler validates request and extracts content
4. Handler creates AnalyzeRequest with content and options
5. Handler calls ML service via gateway.Client.Analyze()
6. ML client sends HTTP POST to Python service at /analyze endpoint
7. Python service processes request and returns AnalyzeResponse
8. ML client decodes response and returns it to handler
9. Handler sends response back to frontend
10. Middleware logs the request/response cycle

VALIDATION AND ERROR HANDLING
-----------------------------
- Input validation: Checks for valid input types
- JSON validation: Ensures proper JSON structure for non-upload requests
- File validation: Ensures file is provided for upload requests
- Service availability: Returns 503 if ML service is unreachable
- Panic recovery: ErrorHandler middleware prevents server crashes
- Status code handling: Properly propagates ML service errors

CONFIGURATION
-------------
- Server port: 8080 (hardcoded, configurable via environment if needed)
- ML service URL: From ML_SERVICE_URL environment variable
  * Defaults to http://localhost:8000 for development
- CORS origins: http://localhost:3000 (frontend dev server)
- Timeout: 30 seconds for ML service requests

COMPATIBILITY WITH OTHER SERVICES
---------------------------------
The backend is designed to work specifically with:
1. Python ML Service (ml-service/):
   - Expects POST /analyze endpoint accepting AnalyzeRequest JSON
   - Returns AnalyzeResponse JSON matching the defined structure
   - Runs on port 8000 (configurable via ML_SERVICE_URL)

2. React Frontend (frontend-react/):
   - Sends requests to http://localhost:8080/analyze
   - Accepts JSON responses matching AnalyzeResponse structure
   - Handles file uploads to http://localhost:8080/analyze/upload
   - Expects CORS headers allowing requests from localhost:3000

TESTING READINESS
-----------------
The implementation includes:
- Structured error handling for all failure modes
- Input validation to prevent malformed requests
- Clear separation of concerns between layers
- Environment-based configuration for deployment flexibility
- Proper HTTP status codes for different outcomes
- Logging for debugging and monitoring

To test the backend:
1. Ensure Python ML service is running on port 8000
2. Set ML_SERVICE_URL environment variable if needed
3. Run: go run cmd/server/main.go
4. Test endpoints:
   - GET http://localhost:8080/health
   - POST http://localhost:8080/analyze with JSON body
   - POST http://localhost:8080/analyze/upload with form data
